# 编译器实验报告规范

本规范从实验1、实验2的报告写作中提炼，适用于后续所有编译器实验。使用 Markdown 格式，Typora 编辑。

## 报告结构

```
# 实验X：实验名称

[TOC]

## 实验内容（必做）
（直接描述任务，不用"本实验要求"开头）

### 设计思路
### 实现过程
### 测试用例说明

## 实验结果（必做）
（截图）

## 实验内容（选做）

### 设计思路
### 实现过程
### 测试用例说明

## 实验结果（选做）
（截图）

## 个人总结
```

注意：
- 顶级标题直接用 `# 实验X：实验名称`，不加课程名外层标题
- 实验内容开头直接描述做什么，语气简洁直接

## 编号与列表

实现过程的步骤用有序列表，正文缩进跟在标题后：

```markdown
1. 文件读取与 DFA 构建
   	程序逐行读取文本文件，依次解析...

2. DFA 合法性检查
   	按照实验要求，程序对读入的 DFA 进行四项检查...
```

## 图片

在对应位置放占位符告诉我需要什么图，需要进行什么操作，由我来截取。

## 文字风格

- 不贴源代码，写设计思路与实现过程
- 不用中文引号框概念做强调（~~这种"分层思路"~~）
- 不用破折号做插入式解释，用逗号断句或另起一句
- 不加多余的括号注释，上下文够理解就省略
- 不用"首先...其次...最后..."模板化过渡
- 不在段落末尾加"便于..."、"有助于..."套话
- 语言直接平实，陈述事实，不刻意修辞
- 行内代码变量、函数名用反引号包裹

## 视角与叙事

- 从"一个学生拿到任务后怎么分析、怎么做"的视角写，不是从"AI 读了 PPT 后总结"的视角
- 不引用 PPT 页码、图片编号（如"PPT 第9页"、"image18"），没有人会这么写报告
- 老师不知道代码长什么样，只看设计思路、实现过程和结果。讲清楚系统是如何设计的，不要逐行翻译代码
- 不把思考链直接写进报告，而是组织成自然的分析过程
- 不做不必要的比较或自我辩护（如"这比 X 方案要简洁得多"），直接写选择了什么方案
- 实验结果部分简洁陈述结论，不重复实验内容已经说过的细节
- 避免主观评价词（如"信心"、"直觉"），用客观表述（如"理解"、"认识"）

## 术语风格

遵循 `/lab-report` skill 的术语规则。要点：
- 并列术语风格统一：一组并列的词要么都用中文，要么都用英文，不要混搭
- 编译原理常用中文术语保留中文：关键字、标识符、界符、接受状态、死状态、状态转移、字符集
- 代码中的定义直接写代码形式（如 `Token` 包含 `kind`、`lexeme[128]`），不翻译成中文描述
- token、lexeme、field 等 `/lab-report` 偏好英文列表中的词直接用英文

## 个人总结

- 150-400 字，不写流水账
- 开头用"本实验"引出，概括做了什么
- 中间展开具体的技术收获和认识加深
- 结尾用"总体而言"（或其他词）收束
